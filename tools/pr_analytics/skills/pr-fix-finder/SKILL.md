# PR Fix Finder

这个 skill 用于回答"某个问题是否已经被历史 PR 修复过"。它只负责编排检索，不负责在本地生成 embedding 或做 PR 相关性推理。

## 核心前提

- **知识库状态**：库中所有的 PR 都是**已合入 (Merged)** 的。不存在"正在修复"或"待合并"的状态。
- **目标**：找到修复该问题的具体 PR，或确认在当前知识库中未发现相关修复。

## 何时使用

- 用户描述一个 StarRocks 问题，想知道是否已有修复 PR。
- 用户给出报错、堆栈、类名、函数名、SQL、模块名，需要回查历史修复。
- 需要给出"已修复 / 未确认 / 未发现"的结论，并附上证据 PR 和版本信息。

## 依赖接口

默认假设 PR 检索服务已经启动，基础地址为 `http://172.26.80.15:8888`。如果用户或环境明确提供了其他地址，优先使用那个地址。

### 接口定义

- `GET /api/agent/search`
  - 语义检索
  - 参数：`query`（推荐）
  - 返回 `{"results": [...]}`
- `GET /api/agent/filter`
  - 关键词检索
  - 参数：`keyword`（推荐）
  - 返回 `{"results": [...]}`
- `GET /api/agent/pr/<number>`
  - PR 详情
  - 返回 `{"result": {...}}`，其中 `versions` 字段为数组，每个元素是该 PR 合入的分支名（如 `["main", "4.0.1", "3.5.2"]`）

### 本地访问限制

访问本地检索服务时必须绕过代理。无论使用 `curl`、`WebFetch` 还是其他 HTTP 工具，都要确保请求不经过 `http_proxy` / `https_proxy`。使用 `curl` 时固定带上 `--noproxy 127.0.0.1,localhost`。

如果目标地址是 `127.0.0.1` 或 `localhost`，默认要意识到这可能是 agent 沙箱自己的回环地址，而不是宿主机的本地服务地址。

执行规则直接简化为：

- 在沙箱里请求本地 PR 检索服务时，只要请求失败，不必继续区分失败原因。
- 失败后立即改用宿主机 / 沙箱外执行同一个请求。
- 如果当前 agent 不能直接切到宿主机执行，就请用户在宿主机执行该请求并提供结果，或者提供一个当前沙箱可访问的服务地址。
- 在宿主机视角下同一个请求仍然失败时，才可视为服务异常；在此之前，不要下“接口不可用”或“没有相关 PR”的结论。

## 工作流

最多做 3 轮，每轮在 agent 内部推理过程中记录：

- 本轮使用的关键词
- 命中的候选 PR
- 为什么保留或排除这些候选
- 下一轮关键词如何重写

最终输出时，将每轮记录压缩为一句话摘要（见"输出要求"）。

### 职责边界

- agent 负责流程编排：决定是否进入下一轮、调用哪个接口、合并结果、何时补 PR 详情、何时停止。
- 外部大模型负责语义处理：从用户问题、候选 PR、PR 详情中提取症状词、术语词、代码符号，并生成下一轮关键词。
- skill 负责约束重写方式，不允许模型随意发散，也不允许 agent 跳过证据直接下结论。

### 检索语言策略

- 每一轮都要同时尝试中文和英文关键词，不要只搜单一语言。
- `query` 和 `keyword` 都按同一轮的核心意图生成中英文两个版本，再分别调用搜索接口。
- 如果用户原问题里已经包含英文报错、类名、函数名、配置项或 SQL，不要把这些词翻译掉；中英文版本都应保留这些高价值 token。
- 中英文结果返回后按 `pr_number` 去重合并，再交给外部大模型判断相关性。
- 控制请求数量：每一轮围绕同一个意图，中文和英文各 1 个版本即可，不要无限扩写同义词。

### 第 1 轮

直接从用户问题中提取：

- 原始报错
- 关键现象
- 类名 / 函数名 / 文件名
- 模块词（FE / BE / MV / Load / Compaction / Query / Tablet 等）

优先调用：

1. `/api/agent/filter`
2. `/api/agent/search`

将两个结果按 `pr_number` 去重合并，再交给外部大模型判断相关性。

### 第 2 轮

如果没有明确命中，由外部大模型基于用户问题和第一轮结果做"术语化重写"。agent 负责把以下信息提供给模型：

- 用户原始问题
- 第一轮使用过的关键词
- 第一轮命中的候选 PR 标题和摘要
- 第一轮的 miss 原因：为什么这些 PR 不够相关

模型按以下规则重写：

- 把通俗描述改成 StarRocks 术语
- 把宽泛现象改成模块 + 症状
- 把长句改成短关键词组合
- 保留高价值符号名，不要把具体报错完全丢掉

模型输出要求：

- 输出 3 到 5 个短 query
- 至少保留 1 个接近用户原话的 query
- 至少生成 1 个"模块 + 症状" query
- 每个核心 query 都要补出对应的英文或中文版本，供 agent 分别填入 `query` 和 `keyword`
- 不要输出空泛词，如 `fix`、`issue`、`problem`

再次调用 `/api/agent/filter` 和 `/api/agent/search`。

### 第 3 轮

如果仍然不清晰，由外部大模型基于前两轮结果做"符号化重写"。agent 负责把以下信息提供给模型：

- 用户给出的 stack trace、类名、函数名、配置项、SQL
- 前两轮高相关候选 PR 的标题、摘要、body
- `/api/agent/pr/<number>` 补到的详情

模型按以下规则重写：

- 类名
- 函数名
- 配置项
- 特定 SQL 功能名

模型输出要求：

- 输出 2 到 4 个高信息量 query
- 优先使用"符号 + 症状"或"符号 + 模块"的组合
- 每个核心 query 都要补出对应的英文或中文版本，供 agent 分别填入 `query` 和 `keyword`
- 丢掉泛词，如 `fix`、`error`、`optimize`
- 如果没有提取到有效符号，不要伪造代码名，直接维持第二轮最好的 query

只要发现高相关候选，就停止继续扩展搜索。

每轮内对 filter 和 search 的调用可以并发执行，但各轮之间必须串行（下一轮依赖上一轮的结果）。

## 候选评估规则

外部大模型必须对每个候选输出以下之一：

- `strong`: PR 很可能直接修复了用户问题
- `possible`: PR 改到了相邻逻辑，但证据还不够
- `irrelevant`: 关键词相似，但问题本质不同

判定 `strong` 时，至少满足其中两项：

- 标题或摘要直接描述了相同问题
- PR body 的 `Why I'm doing` / `What I'm doing` 与用户现象一致
- 涉及相同模块、相同代码符号或相同功能路径

## 详情补全

如果 `/api/agent/search` 或 `/api/agent/filter` 返回的信息还不够，就调用：

- `/api/agent/pr/<number>` 获取完整 body 和版本信息

如果仍然不够，再使用仓库工具补充：

- `gh pr view <number> --json title,body,files,commits`
- `gh pr diff <number>`

不要把 `gh` / `git` 作为第一步；只对高度怀疑的候选做详情补全，最多补全 6 个。

## 停止条件

- 发现 `strong` 候选后停止
- 3 轮后仍然没有 `strong`，输出"未发现明确修复"
- 如果只有 `possible`，明确说明证据不足，不要强行下结论

## 输出要求

最终回答必须包含：

- `✅/❌/❓ 结论`：`✅ 已修复` / `❌ 未发现明确修复` / `❓可能相关但证据不足`
- `🔗 证据 PR`：编号、标题、理由
- `🌿 版本信息`：从 PR 详情的 `versions` 数组推断修复进入了哪些分支（如 `versions: ["main", "3.3.2"]` 表示修复已合入 main 分支和 3.3.2 版本）
- `🔎 检索过程摘要`：最多 3 轮，每轮一句话

## 约束

- 不要虚构未命中的 PR，不要声称某个修复"应该存在"。
